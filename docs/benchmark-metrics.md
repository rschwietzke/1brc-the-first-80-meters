# Benchmark Execution & Metrics Collection

This document details exactly what happens during a benchmark run for a given Java test class, and explains the full suite of metrics captured by the 1BRC reporting harness.

## 1. What Happens During a Run?

When you execute `./benchmark-matrix.sh run`, the system does not run a simple generic loop. Instead, it generates a highly specific execution matrix based on your `benchmark.conf`. 

For *every single* Java implementation class (e.g., `BRC001_Baseline`, `BRC042_NioChunking`), the harness performs a dedicated execution for *each* valid permutation of your configuration variables. 

A single "Run Permutation" consists of:
1. **JVM Spin-up**: The harness boots a completely fresh JVM using the specific `JDK`, `GC_OPTS`, and `VM_OPTS` defined for that permutation.
2. **Resource Constraints**: If a `TASKSET` is defined (e.g., locking the run to 8 CPU cores), the Linux `taskset` utility is applied to the JVM process.
3. **Hardware Telemetry Profiling**: The execution is wrapped in the Linux `perf stat` utility and the GNU `/usr/bin/time` utility to capture deep hardware counters from the CPU.
4. **JFR Recording (Optional)**: If `--jfr` is passed, the JVM enables Java Flight Recorder to trace deep internal JVM events (like GC cycles and object allocations).
5. **Execution & Checksum Output**: The Java class executes the 1BRC challenge against the specified `DATASET`. The class is responsible for printing its final execution time and calculated Checksum to standard output.

## 2. What Metrics Are Captured?

The harness merges data from three completely different sources (Standard Output, `perf stat`, and Java Flight Recorder) into a single, unified CSV row for every permutation. 

### A. Core Application Metrics (Standard Output)
These metrics are explicitly printed by the Java class itself at the end of the run:
*   **Median Runtime (ms)**: The total time the Java implementation took to process the dataset.
*   **Checksum**: The final calculated hash of the 1BRC temperature data. The harness automatically compares this against the designated "Baseline" implementation to generate a Pass/Fail validation in the HTML reports.

### B. Hardware Telemetry Metrics (`perf stat`)
These metrics represent the raw physical execution on the CPU:
*   **Instructions**: The total number of machine-level instructions executed by the CPU. Dropping instructions (e.g., via SWAR techniques) is the primary way to gain performance.
*   **Cycles**: The total number of CPU clock cycles consumed.
*   **IPC (Instructions Per Cycle)**: The efficiency of the CPU pipeline. A higher IPC (e.g., > 2.5) means the CPU is successfully executing multiple instructions in parallel per clock tick.
*   **Branches & Branch Misses**: The number of logic branches the code took, and how often the CPU's branch predictor guessed incorrectly. High branch misses severely degrade IPC.
*   **Context Switches & CPU Migrations**: Measures how often the OS had to swap threads out of cores or move them between physical cores. High numbers indicate heavy thread contention.
*   **Time (User / Sys / Elapsed)**: GNU `time` captures how much total time was spent running application code vs operating system kernel code.

### C. JVM Profiling Metrics (Java Flight Recorder)
If the `--jfr` flag is used, `CsvMerger.java` will asynchronously parse the binary `.jfr` file to extract JVM-internal metrics:
*   **GC Pause Time (ms)**: The total amount of time the application was paused by the Garbage Collector.
*   **Allocated Bytes**: The exact number of bytes allocated on the heap during the run. Zero-allocation implementations are highly prized in 1BRC.
*   **JIT Compilation Time (ms)**: The amount of time the JVM's C1/C2 compilers spent optimizing the bytecode into native machine code.

## 3. Ground Truth Validation

The benchmarking suite does not blindly trust the numbers. It employs an automated **Ground Truth Validation** system:

1. The `HtmlReportWriter` actively scans the `src/main/java/` directory looking for the `// status: baseline` annotation.
2. It extracts the Checksum generated by that specific baseline class for a given Dataset/Environment permutation.
3. Every other class executed in that permutation is strictly compared against the baseline's checksum.
4. The HTML Matrix reports will automatically flag each run as `[Pass]`, `[Fail]`, or `[Skip: Known Incorrect]` (for classes tagged with `// status: incomplete` or `// @Attention`).
